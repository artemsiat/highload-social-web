# План выполнения ДЗ 3: Репликация

## Цель плана

Сделать ДЗ поэтапно и без лишнего риска:

1. сначала привести проект в удобное состояние для тестов;
2. затем снять честный baseline на одной БД;
3. только после этого переходить к `primary + 2 replicas` и репликации.

Отдельная цель этого плана: сохранить текущий рабочий single-DB сценарий и не ломать его при переходе к репликации.

---

## Базовая идея по конфигурации

Для этого ДЗ вводим два Spring-профиля:

- `standard-db` — обычная одна PostgreSQL;
- `primary-replica-db` — `primary + replica1 + replica2`.

### Почему это хорошая схема

- сохраняем текущий рабочий режим;
- не смешиваем single-node и replication-конфигурацию в одном файле;
- проще сравнивать baseline и replicated setup;
- проще откатиться к исходному состоянию;
- проще объяснить проверяющему, как запускать проект в двух режимах.

### Как будет выглядеть структура конфигов

#### `application.yaml`

В основном файле оставляем только общие настройки:

- `spring.application.name`
- `management.*`
- `server.port`
- `application.test-data.*`
- общие настройки `docker compose`, если они не завязаны на конкретную БД
- активный профиль по умолчанию

Идея такая:

- в `application.yaml` не храним datasource для конкретного режима;
- там задаем профиль по умолчанию, чтобы исходный сценарий сохранился.

Практически это означает:

- по умолчанию активен `standard-db`;
- при необходимости можно переключиться на `primary-replica-db`.

#### `application-standard-db.yaml`

Здесь будет все, что относится к обычной single-DB схеме:

- `spring.datasource.*`
- `spring.flyway.*`
- при необходимости `spring.docker.compose.*`, если single-node запуск будет иметь свои особенности

Это станет прямым продолжением текущего рабочего режима.

#### `application-primary-replica-db.yaml`

Здесь будет все, что относится к репликации:

- datasource для `primary`
- datasource для `replica1`
- datasource для `replica2`
- routing datasource
- настройки Flyway только для primary
- при необходимости отдельные параметры для read/write split

### Что считаем дефолтным поведением

Чтобы сохранить исходную схему, по умолчанию проект должен подниматься в профиле `standard-db`.

То есть:

- обычный запуск приложения работает как и сейчас;
- профиль репликации включается отдельно и осознанно;
- основной `application.yaml` выступает как точка входа, а не как место со всеми сразу DB-настройками.

### Небольшое замечание по названию

Я бы использовал имя `standard-db`, а не `standart-db`, чтобы не тащить опечатку в названия файлов, профилей и переменных окружения.

## Этап 1. Рефакторинг и автогенерация тестовых данных

### Что хотим получить

Сейчас тестовые данные генерируются через API. Нужно добавить второй сценарий:

- генерация при старте приложения;
- количество записей для старта задается отдельным конфигом;
- startup-логика ориентируется только на `startup-record-count`;
- лимит для REST API задается отдельно через `max-allowed-records`;
- если в таблице уже есть `startup-record-count` записей или больше, стартовая генерация не должна запускаться.

### Конфигурация

Настройки держим в общем конфиге, потому что они нужны в обоих DB-режимах:

```yaml
application:
  test-data:
    generate-on-startup: true
    startup-record-count: 1000000
    max-allowed-records: 5000000
```

### Что изменить

1. Вынести startup generation в отдельный startup-компонент.
2. Переиспользовать текущую бизнес-логику генерации, а не дублировать ее.
3. Развести два сценария:
   - startup generation смотрит только на `startup-record-count`;
   - REST API generation ограничивается только `max-allowed-records`.
4. Сделать startup generation идемпотентной:
   - проверить текущее количество строк;
   - вычислить, сколько еще нужно дозаписать;
   - никогда не превышать `startup-record-count`.
5. Сделать логи явными:
   - текущий count;
   - target count;
   - actual generated count;
   - причина пропуска генерации.

### Критерий готовности

- приложение при пустой таблице дозаполняет ее до `startup-record-count`;
- повторный старт не создает лишние записи;
- REST API генерации продолжает работать;
- REST API не превышает `max-allowed-records`.

## Этап 2. Переход на профильную конфигурацию БД

### Зачем

Сейчас datasource живет в основном `application.yaml`. Для ДЗ с репликацией это начинает мешать:

- single-node и replication-конфигурация смешиваются;
- сложнее сохранить исходный сценарий;
- сложнее понять, какие свойства относятся к какому режиму.

### Что хотим получить

- `application.yaml` содержит только общие настройки;
- `application-standard-db.yaml` содержит single-node datasource;
- `application-primary-replica-db.yaml` содержит replication-конфигурацию;
- дефолтный профиль — `standard-db`.

### Что переносим в профиль `standard-db`

- `spring.datasource.url`
- `spring.datasource.username`
- `spring.datasource.password`
- `spring.datasource.hikari.schema`
- `spring.flyway.enabled`
- `spring.flyway.locations`
- `spring.flyway.default-schema`

### Что закладываем в профиль `primary-replica-db`

На этом этапе можно еще не реализовывать routing, но уже зафиксировать структуру:

- параметры primary datasource;
- параметры replica1 datasource;
- параметры replica2 datasource;
- отдельная зона для routing-настроек;
- Flyway только через primary.

### Важное проектное решение

Основной сценарий должен оставаться таким:

- приложение стартует без специальных флагов;
- активируется `standard-db`;
- все продолжает работать так же, как до подготовки к репликации.

А сценарий репликации:

- включается отдельно;
- не ломает исходную конфигурацию;
- не влияет на baseline single-node.

### Критерий готовности

- у проекта есть один основной `application.yaml` без конкретной DB-схемы подключения;
- есть `application-standard-db.yaml`;
- есть `application-primary-replica-db.yaml`;
- дефолтный запуск использует `standard-db`.

## Этап 3. Docker PostgreSQL как основной runtime для `standard-db`

### Зачем

Для baseline и для отчета нужен воспроизводимый способ запуска. Поэтому single-node режим должен работать через Docker PostgreSQL, а не зависеть от локального PostgreSQL в macOS.

### Что хотим получить

- профиль `standard-db` использует одну PostgreSQL в Docker;
- Docker Compose становится основным способом поднятия single-node БД;
- локальный PostgreSQL в macOS перестает быть обязательной частью окружения.

### Что изменить

1. Проверить, что `compose.yaml` стабильно поднимает одну PostgreSQL.
2. Убедиться, что `standard-db` использует именно этот контейнер.
3. Привести документацию к одному сценарию запуска:
   - одна БД в Docker;
   - приложение поверх нее.
4. Проверить Flyway и startup generation на Docker-БД.

### Критерий готовности

- `standard-db` стабильно стартует против Docker PostgreSQL;
- миграции проходят;
- тестовые данные генерируются;
- сценарий запуска воспроизводим.

## Этап 4. Baseline на одной БД

### Почему это обязательно

До репликации нам нужно понять, как система ведет себя в single-node режиме. Иначе потом невозможно честно сравнивать результат.

### Что измеряем

#### Чтение

- `GET /user/get/{id}`
- `GET /user/search`
- mixed-read сценарий как более реалистичная комбинация обоих endpoint-ов

#### Запись

- `POST /user/register`
- либо отдельный lightweight write endpoint для будущего failover-теста

### Что подготовить

1. Заполнить таблицу `users` до `1_000_000` записей.
2. Подготовить сценарии `k6`:
   - `user-get`
   - `user-search`
   - `mixed-read`
3. Сохранить JSON-результаты.
4. Сохранить Grafana-скриншоты.

### Какие метрики собрать

- latency: `avg`, `p90`, `p95`, `p99`, `max`;
- throughput;
- error rate;
- CPU;
- память;
- пул соединений;
- при возможности метрики PostgreSQL.

### Критерий готовности

- есть воспроизводимый baseline в профиле `standard-db`;
- есть артефакты для отчета;
- понятно, что именно будет сравниваться с replicated setup.

## Этап 5. Подготовка приложения к read/write split

### Что хотим получить

Приложение должно уметь:

- отправлять `read-only` транзакции на replica;
- отправлять write-транзакции на primary.

### Что менять в коде

1. Ввести отдельную конфигурацию datasource:
   - `primary`;
   - `replica-1`;
   - `replica-2`.
2. Добавить routing datasource на основе `AbstractRoutingDataSource`.
3. Использовать `TransactionSynchronizationManager.isCurrentTransactionReadOnly()`.
4. Обернуть routing datasource в `LazyConnectionDataSourceProxy`.
5. Явно отметить сервисные методы:
   - чтение: `@Transactional(readOnly = true)`
   - запись: `@Transactional`

### Где это важно в текущем проекте

- `search(...)` в `UserService` — read only;
- `getProfileById(...)` в `UserService` — read only;
- `register(...)` в `UserService` — write;
- позже тестовый write endpoint — write.

### Дополнительно

- Flyway должен идти только в primary;
- полезно логировать выбранный datasource;
- полезно различать подключения через `application_name`.

### Критерий готовности

- `primary-replica-db` технически готов к read/write split;
- `standard-db` не ломается от этих изменений;
- single-node режим по-прежнему остается рабочим дефолтом.

## Этап 6. Настройка `1 primary + 2 replicas`

### Что хотим получить

Рабочую streaming replication:

- `primary`
- `replica1`
- `replica2`

### Как лучше делать

Использовать Docker Compose как основной способ поднятия topology.

Причины:

- проще поднять несколько изолированных узлов;
- проще менять конфиг PostgreSQL;
- проще воспроизводить проверяющему;
- проще выполнять `stop`, `kill`, `promote`.

### Что нужно настроить в PostgreSQL

- `wal_level=replica`
- `max_wal_senders`
- `max_replication_slots`
- `wal_keep_size`
- replication user
- `pg_hba.conf` для replica connections
- `pg_basebackup` для инициализации реплик

### Что проверить

- реплики догоняют primary;
- `pg_stat_replication` показывает обе реплики;
- `pg_is_in_recovery()`:
  - `false` на primary
  - `true` на replica

### Критерий готовности

- две реплики стабильно получают изменения;
- приложение может подключаться к primary и replica-раздельно;
- replication-сценарий живет отдельно от `standard-db`.

## Этап 7. Нагрузочное тестирование после read/write split

### Что делаем

Повторяем baseline-тесты, но уже в профиле `primary-replica-db`.

### Что важно доказать

Не только изменение latency, но и сам факт, что чтение ушло с primary.

### Чем доказать

- `pg_stat_activity` на primary и replica;
- `pg_stat_replication`;
- логи приложения с выбранным datasource;
- графики HikariCP / HTTP / PostgreSQL;
- при необходимости SQL-счетчики по узлам.

### Возможный ожидаемый результат

На одной машине latency может улучшиться не драматически. Это нормально.

Для зачета важнее показать:

- чтение действительно идет на replica;
- primary разгружен от read-only запросов;
- архитектурное разделение работает корректно.

### Критерий готовности

- есть таблица `before / after`;
- есть доказательство переноса чтения на replica;
- есть комментарий по эффекту на производительность.

## Этап 8. Quorum synchronous replication

### Что хотим получить

Настроить режим, в котором commit подтверждается только после подтверждения хотя бы одной реплики.

### Базовый вариант

- `synchronous_standby_names = 'ANY 1 (replica1, replica2)'`

### Что нужно проверить

- write latency выросла по сравнению с asynchronous replication;
- replica действительно участвует в подтверждении commit;
- при аварии primary подтвержденные транзакции не теряются.

### Критерий готовности

- включен synchronous quorum режим;
- есть понятный набор замеров до и после.

## Этап 9. Эксперимент с потерей / непотерей транзакций

### Что лучше сделать в проекте

Добавить отдельную тестовую таблицу и простой endpoint для записи, чтобы эксперимент был контролируемым.

Пример идеи:

- таблица `replication_test_events`
- endpoint `POST /replication-test/write`

Каждый успешный ответ должен однозначно означать:

- одна запись была committed в БД.

### Сценарий эксперимента

1. Запустить постоянную write-нагрузку.
2. На стороне клиента считать число успешных ответов.
3. Остановить primary.
4. Завершить нагрузку.
5. Выбрать самую свежую replica.
6. Промоутить ее в новый primary.
7. Перенастроить оставшуюся replica.
8. Сравнить:
   - сколько write-запросов клиент считает успешными;
   - сколько строк реально есть в новой primary.

### Что должно получиться

В синхронном режиме подтвержденные клиенту транзакции не должны потеряться.

### Критерий готовности

- есть воспроизводимый failover-сценарий;
- есть численное сравнение `successful writes` vs `rows in promoted primary`;
- есть вывод о наличии или отсутствии потерь.
