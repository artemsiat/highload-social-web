# План выполнения ДЗ 3: Репликация

## Цель плана

Сделать работу по ДЗ поэтапно и без лишних скачков:

1. сначала привести проект в удобное состояние;
2. затем получить честный baseline на одной БД;
3. только после этого переходить к master/slave и синхронной репликации.

Такой порядок даст нам:

- меньше смешанных изменений в одном PR/коммите;
- понятное сравнение `single DB` против `replicated DB`;
- более убедительный отчет по ДЗ.

---

## Общая стратегия

### Этап 1. Рефакторинг и подготовка данных

Сначала улучшаем генерацию тестовых данных, чтобы проект сам поднимался в удобном для тестов состоянии.

### Этап 2. Переход на Docker PostgreSQL как основной runtime

Дальше приводим проект к сценарию, где основная БД для запуска и тестов работает в Docker, а не на локальном PostgreSQL в macOS.

### Этап 3. Baseline на одной БД

Перед репликацией фиксируем производительность на одной инстанции БД:

- `/user/search`
- `/user/get/{id}`
- нагрузка на чтение
- отдельно нагрузка на запись

### Этап 4. Репликация и routing read/write

После baseline настраиваем `1 master + 2 slave`, переносим чтение на реплики, запись оставляем на master.

### Этап 5. Синхронная репликация и failover-эксперимент

После обычной streaming replication добавляем quorum synchronous replication и проверяем потерю/непотерю транзакций при падении master.

### Этап 6. Отчет

В конце собираем артефакты, графики, таблицы, выводы и оформляем финальный отчет.

## Этап 1. Рефакторинг и автогенерация тестовых данных

### Что хотим получить

Сейчас тестовые данные генерируются только через API. Нужно добавить второй сценарий:

- генерация при старте приложения;
- количество записей для старта задается через отдельный конфиг;
- startup-логика ориентируется только на `startup-record-count`;
- лимит для REST API задается отдельно через `max-allowed-records`;
- если в таблице уже есть `startup-record-count` записей или больше, стартовая генерация не должна запускаться.

### Что изменить

1. Вынести параметр количества стартовых записей в конфиг.
2. Добавить отдельную настройку, например:

```yaml
application:
  test-data:
    generate-on-startup: true
    startup-record-count: 1000000
    max-allowed-records: 5000000
```

3. Добавить startup-инициализацию через `ApplicationRunner` или `CommandLineRunner`.
4. Переиспользовать текущую логику `TestDataService`, а не дублировать ее.
5. Развести правила для двух сценариев:
   - startup generation смотрит только на `startup-record-count`;
   - REST API generation ограничивается только `max-allowed-records`.
6. Сделать startup generation идемпотентной:
   - проверить текущее количество строк;
   - вычислить, сколько еще нужно дозаписать до `startup-record-count`;
   - записывать только недостающее число;
   - никогда не превышать `startup-record-count`.
7. Для REST API оставить независимый лимит:
   - ручная генерация может дозаполнять данные сверх стартового объема;
   - но никогда не должна превышать `max-allowed-records`.
8. Логи сделать явными:
   - текущий count;
   - requested count;
   - actual generated count;
   - причина, почему генерация пропущена.

### Дополнительный рефакторинг в этом этапе

- разделить понятия:
  - ручная генерация через API;
  - генерация при старте;
  - общая внутренняя бизнес-логика генерации;
- явно разделить два ограничения:
  - целевой объем для startup;
  - максимальный объем для REST API;
- проверить, что batch insert и лимиты живут в одном месте;
- не завязывать startup-инициализацию на web-слой.

### Критерий готовности

- приложение стартует и при пустой таблице дозаполняет ее до `startup-record-count`;
- повторный старт не создает записей сверх `startup-record-count`;
- API генерации продолжает работать;
- REST API не превышает `max-allowed-records`.

## Этап 2. Переход на Docker PostgreSQL как основной способ запуска БД

### Зачем

Сейчас у проекта смешанный режим: часть настроек ориентирована на Docker Compose, но фактически в работе используется локальный PostgreSQL на macOS. Для ДЗ лучше сделать один основной и воспроизводимый путь: БД в Docker.

### Что хотим получить

- приложение по умолчанию работает с PostgreSQL в Docker;
- Docker Compose становится основным способом поднятия БД;
- конфигурация подключения становится прозрачной и предсказуемой.

### Что изменить

1. Пересмотреть дефолтные значения `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`.
2. Убедиться, что приложение стабильно стартует вместе с Docker PostgreSQL и не зависит от локального PostgreSQL в macOS.
3. Обновить `README`:
   - основной сценарий: Docker PostgreSQL;
   - локальный PostgreSQL оставить только как optional path, если вообще нужен.
4. Проверить миграции Flyway на Docker-БД.
5. Подготовить понятный сценарий запуска:
   - БД `social`;
   - пользователь;
   - пароль;
   - порт;
   - volume для данных.

### Практический подход

Для baseline и ранней разработки использовать один PostgreSQL-контейнер в Docker.

Для multi-node replication позже расширить тот же Docker Compose до:

- `primary`
- `replica1`
- `replica2`

То есть план такой:

- `single DB baseline` на одном Docker PostgreSQL;
- затем расширение до `replication topology` в Docker Compose.

### Критерий готовности

- приложение стабильно стартует с Docker PostgreSQL;
- миграции проходят;
- генерация тестовых данных работает на Docker-БД;
- документация отражает Docker как основной путь запуска.

## Этап 3. Baseline на одной БД

### Почему это обязательно

До master/slave нам нужно понять, как система ведет себя на одной БД. Иначе потом нельзя будет честно сказать, что улучшилось, а что просто изменилось.

### Что измеряем

#### Чтение

- `GET /user/get/{id}`
- `GET /user/search`

#### Запись

- `POST /user/register`
- либо отдельный lightweight write endpoint для будущего failover-теста

### Что подготовить

1. Заполнить таблицу `users` до `1_000_000` записей.
2. Подготовить набор реальных `id` для `/user/get/{id}`.
3. Переиспользовать текущий k6-сценарий для `/user/search`.
4. Добавить новый k6-сценарий для `/user/get/{id}`.
5. Добавить сценарий записи:
   - либо на `/user/register`;
   - либо на специальную тестовую таблицу, что позже лучше подойдет для failover-эксперимента.

### Какие метрики собрать

- latency: `avg`, `p90`, `p95`, `p99`, `max`;
- throughput;
- error rate;
- CPU;
- память;
- пул соединений;
- по возможности метрики PostgreSQL:
  - active sessions;
  - transactions;
  - cache hit ratio;
  - slow queries.

### Какие прогоны сделать

Для чтения:

- `1 VU`
- `10 VU`
- `100 VU`
- `1000 VU`

Для записи:

- более умеренные значения, например `1 / 10 / 50 / 100`, чтобы не превратить эксперимент в шум.

### Что сохранить

- JSON-результаты `k6`;
- таблицу с итогами;
- графики `latency` и `throughput`;
- короткий текстовый вывод по single-node режиму.

### Критерий готовности

- у нас есть воспроизводимый baseline для одной БД;
- понятно, какие endpoint-ы являются read-heavy;
- есть цифры, с которыми будем сравнивать master/slave.

## Этап 4. Подготовка приложения к read/write split

### Что хотим получить

Приложение должно уметь:

- отправлять `read-only` транзакции на slave;
- отправлять write-транзакции на master.

### Что менять в коде

1. Ввести отдельную конфигурацию datasource:
   - `master`;
   - `slave-1`;
   - `slave-2`.
2. Добавить routing datasource на основе `AbstractRoutingDataSource`.
3. Использовать `TransactionSynchronizationManager.isCurrentTransactionReadOnly()`.
4. Обернуть routing datasource в `LazyConnectionDataSourceProxy`, чтобы выбор происходил после открытия транзакции.
5. Явно отметить сервисные методы:
   - чтение: `@Transactional(readOnly = true)`
   - запись: `@Transactional`

### Где именно в текущем проекте это важно

- `search(...)` в `UserService` — read only;
- `getProfileById(...)` в `UserService` — read only;
- `register(...)` в `UserService` — write;
- позже тестовый write endpoint — write.

### Дополнительно

- Flyway должен идти только в master;
- логировать, какой datasource был выбран;
- по возможности различать подключения через `application_name`.

### Критерий готовности

- на single-node режиме routing-конфигурация не ломает проект;
- read-only методы технически готовы к переносу на реплики.

## Этап 5. Настройка `1 master + 2 slave`

### Что хотим получить

Рабочую streaming replication:

- `primary`
- `replica1`
- `replica2`

### Как лучше делать

Для этого этапа разумно использовать тот же Docker Compose, который уже будет основным способом запуска БД после этапа 2.

Причины:

- проще поднять несколько изолированных узлов;
- проще менять конфиг PostgreSQL;
- проще воспроизводить проверяющему;
- проще выполнить `kill`, `stop`, `promote`.

### Что нужно настроить в PostgreSQL

- `wal_level=replica`
- `max_wal_senders`
- `max_replication_slots`
- `wal_keep_size`
- replication user
- `pg_hba.conf` для replica connections
- `pg_basebackup` для инициализации реплик

### Что проверить

- реплики догоняют master;
- `pg_stat_replication` показывает обе реплики;
- `pg_is_in_recovery()`:
  - `false` на master
  - `true` на slave

### Критерий готовности

- две реплики стабильно получают изменения;
- приложение может подключаться к master/slave-нодам раздельно.

## Этап 6. Нагрузочное тестирование после read/write split

### Что делаем

Повторяем baseline-тесты, но уже после внедрения routing на реплики.

### Что важно доказать

Не только изменение latency, но и сам факт, что чтение ушло со master.

### Чем доказать

- `pg_stat_activity` на master и replica;
- `pg_stat_replication`;
- логи приложения с выбранным datasource;
- графики HikariCP / HTTP / PostgreSQL;
- при необходимости SQL-счетчики по узлам.

### Возможный ожидаемый результат

На одной машине latency может улучшиться не драматически. Это нормально.

Для зачета важнее показать:

- чтение действительно идет на slave;
- master разгружен от read-only запросов;
- архитектурное разделение работает корректно.

### Критерий готовности

- есть таблица `before / after`;
- есть доказательство переноса чтения на slave;
- есть комментарий по эффекту на производительность.

## Этап 7. Quorum synchronous replication

### Что хотим получить

Настроить режим, в котором commit подтверждается только после подтверждения хотя бы одной реплики.

### Базовый вариант

- `synchronous_standby_names = 'ANY 1 (replica1, replica2)'`

### Что нужно проверить

- write latency выросла по сравнению с asynchronous replication;
- реплика действительно участвует в подтверждении commit;
- при аварии master подтвержденные транзакции не теряются.

### Критерий готовности

- включен synchronous quorum режим;
- есть понятный набор замеров до и после.

## Этап 8. Эксперимент с потерей / непотерей транзакций

### Что лучше сделать в проекте

Добавить отдельную тестовую таблицу и простой endpoint для записи, чтобы эксперимент был контролируемым.

Пример идеи:

- таблица `replication_test_events`
- endpoint `POST /replication-test/write`

Каждый успешный ответ должен однозначно означать:

- одна запись была committed в БД.

### Сценарий эксперимента

1. Запустить постоянную write-нагрузку.
2. На стороне клиента считать число успешных ответов.
3. Остановить master:
   - `docker stop`
   - или аналог аварийного сценария
4. Завершить нагрузку.
5. Выбрать самую свежую реплику.
6. Промоутить ее в новый master.
7. Перенастроить оставшуюся реплику.
8. Сравнить:
   - сколько write-запросов клиент считает успешными;
   - сколько строк реально есть в новой primary.

### Что должно получиться

В синхронном режиме подтвержденные клиенту транзакции не должны потеряться.

### Критерий готовности

- есть воспроизводимый failover-сценарий;
- есть численное сравнение `successful writes` vs `rows in promoted master`;
- есть вывод о наличии или отсутствии потерь.
