# Домашнее задание: Репликация

## Цель

В результате выполнения ДЗ вы настроите разные виды репликации, протестируете их влияние на производительность системы и убедитесь, что в случае синхронной репликации вы не теряете данные при аварии.

## В данном задании тренируются навыки

- обеспечение отказоустойчивости проекта;
- администрирование PostgreSQL;
- настройка репликации;
- проведение нагрузочных тестов.

## Описание

Инструкция к ДЗ также доступна по ссылке.

## Пошаговая инструкция выполнения домашнего задания

1. Выберите 2 запроса на чтение: `/user/get/{id}` и `/user/search` из спецификации.
2. Составьте план нагрузочного тестирования, который отправляет запросы на эти API.
3. Создайте нагрузку на чтение с помощью составленного на предыдущем шаге плана и сделайте замеры.
4. Настройте 2 слейва и 1 мастер. Включите потоковую репликацию.
5. Настройте backend-приложение на работу с реплицированной БД:
   1. Перенесите запросы на чтение (`transaction == read only`) на чтение со слейва.
   2. Перенесите запросы на запись на мастер.
6. Для `Java`/`Spring`-стека можно рассмотреть несколько вариантов реализации:
   - использование `MaxScale` и `MySQL Replication JDBC Driver`;
   - использование database proxy server, например `MySQL Proxy`;
   - использование `Replication DataSource`: `kwon37xi/replication-datasource-boot`.

   Рекомендуется последний вариант: написать `ReplicationRoutingDataSource`, который выбирает `master` или `slave` в зависимости от типа транзакции.

   Данное решение работает с `TransactionSynchronizationManager`.

   При использовании Spring Framework достаточно установить `@Transactional(readOnly = true|false)`.
7. Создайте нагрузку на чтение с помощью составленного на предыдущем шаге плана, сделайте замеры и добавьте сравнение результатов в отчет.
8. Настройте кворумную синхронную репликацию.
9. Опционально воспользуйтесь одним из кластерных расширений Postgres, например `patroni`.
10. Создайте нагрузку на запись в любую тестовую таблицу. На стороне, которой нагружаете, считайте, сколько строк вы успешно записали.

    Например, через API. Для анализа можно использовать дашборды Grafana.

11. Убейте мастер-узел: `kill -9` или `docker stop`.
12. Закончите нагрузку на запись.
13. Выберите самый свежий слейв, промоутьте его до мастера и переключите на него второй слейв.
14. Проверьте, есть ли потери транзакций.

## Форма сдачи ДЗ

- предоставить ссылку на исходный код (`GitHub`, `GitLab` и т.д.);
- предоставить докеризированное приложение, которое можно запустить при помощи `docker-compose` (может лежать рядом с исходным кодом), ИЛИ развернутое приложение, доступное извне, ИЛИ инструкцию по запуску;
- предоставить отчет в свободной форме о проделанной работе.

## Критерии оценки

Оценка происходит по принципу зачет/незачет.

### Требования

- в отчете корректно описано, как настроена репликация;
- 2 запроса переведены на чтение со слейва;
- нагрузочное тестирование показало, что нагрузка перешла на слейв;
- проведен эксперимент по потере и непотере транзакций при аварийной остановке `master`.
